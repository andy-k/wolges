THE ENDGAME SOLVER: EXACT PLAY ONCE THE BAG IS EMPTY

Every leave table in this repo (lab/leavegen.txt) values a KEPT rack by an
estimate, because the rest of the game is still unknown -- more tiles will be
drawn, the opponent's rack is hidden. Once the tile bag is empty, neither of
those is true any more: every tile still in play is either on the board or on a
rack, so with both racks known the whole rest of the game is just a finite,
fully-visible sequence of plays and passes. Leave values stop applying at that
point -- lab/leavegen.txt notes that with an empty bag no tiles are drawn, so
the leave table is not consulted there; this file is about solving that fully-
visible tail EXACTLY, by negamax game-tree search (search every legal reply, at
each ply picking whichever line values best for the player to move, alternating
who that is) with iterative deepening (search one ply deeper each pass, using
the previous pass's answer to speed the next one up) all the way to the game's
end.

evaluate(player_idx) is the entry point that prints the search's per-depth
progress. Callers that want the answer as DATA rather than printed lines get
two additions here: solve(player_idx) runs the identical search silently and
returns just the final converged value, and collect_pv(player_idx, out) walks
the best line the search found and pushes each ply's (equity, play) pair into a
caller-owned, reusable vector, so the caller can keep that line around after
the solver itself goes out of scope. Both share the same core loop as
evaluate() -- pulled out into one private routine with a verbose flag -- so
this is a pure refactor: no search logic changes, and evaluate()'s printed
output is exactly as before.


A REAL VALUE FOR THE DEPTH-LIMIT LEAF

Iterative deepening stops each pass at a fixed depth and returns whatever value
the search assigned to the positions at that depth limit -- the "leaves" of
that pass's search tree (a tree-search leaf; see lab/leavegen.txt for the very
different sense of "leave", the kept tiles after a play). Until now a
depth-limited leaf returned a placeholder zero rather than a real value. It now
returns the leftover-tile point margin the position would score if the game
ended right there (both players passing), from the side to move's perspective
-- the same kind of terminal scoring the rest of the solver already uses when
the game genuinely does end.

Today's converged answer does not change: the search always deepens past a
depth-limited line before it actually stops there (an existing guard already
guarantees the search keeps going rather than settling for a capped-depth
value), so no real search run today ever returns this placeholder as its final
answer. But a depth-limited leaf is still a real, reachable position, not a
sentinel that should never be valued -- it deserves its correct value
regardless of whether today's search ever stops there.


BATCH-SOLVING HARVESTED POSITIONS

autoplay can now harvest the bag-empty positions it plays through into a file.
An opt-in batch mode (set WOLGES_ENDGAME_BATCH to that file's path) reads the
harvested positions back, prunes each one's word list to what is reachable,
solves each with solve() (silent and deterministic, from the previous entry),
and prints only each line's index and its value -- nothing else, so a whole
run's output can be hashed or diffed byte for byte. That makes it easy to
bulk-time the solver over many real positions, or to catch an accidental value
change in one pass. With the env var unset, the existing hardcoded demo
positions run exactly as before.


TWO SEARCH SPEEDUPS

Alpha-beta search (negamax with a window of values it can stop early outside)
prunes a branch as soon as it proves that branch cannot beat what the search
has already found elsewhere. Both speedups here only change how fast the search
reaches the SAME answer, never what that answer is -- alpha-beta returns the
same value no matter how much of the tree it prunes away.

Iterative deepening had been re-searching every depth from scratch with the
widest possible window ((negative infinity, positive infinity)). Aspiration
windows narrow that: from the second depth on, search first in a narrow band
around the PREVIOUS depth's value, since the true value almost always lands
close to it, which lets alpha-beta prune far more of the tree; only if the
narrow search's result actually falls outside that band does the search widen
and try again, so the final value is exactly what the full window would have
found -- a modest wall-clock win, larger the deeper a position's search goes.

Move ordering is the order a search tries a node's legal moves in -- trying the
best ones first lets alpha-beta prune the rest sooner, so most nodes stop after
a handful of moves once a cutoff fires. That makes fully sorting every node's
whole move list on every visit look wasteful: lazy move ordering brings only
the best few moves to the front and sorts the remainder only if the search
actually runs past them. This is value-identical by construction -- alpha-beta
returns the same result regardless of the order moves are tried in -- but for
bag-empty endgames it turns out to be a slowdown, not a speedup: the move lists
are short and iterative deepening revisits each node many times, so the full
sort is already cheap and the extra partial-select-then-tail-sort step is pure
overhead, about 7% slower here at every cutoff point tried. It ships off by
default behind a const, kept as an opt-in study lever for wider or deeper
position mixes where trying fewer moves first might still pay off.


A ZERO-REALLOCATION SCRATCH BUFFER

One of the search's scratch buffers (the placed-tile list, rebuilt every time a
candidate play turns into a new search state) can never hold more than one tile
per board square, since a play only ever adds tiles to empty squares. Reserving
that worst-case capacity once, when the work buffer is first created, means the
search never grows the buffer during a run -- not even on the very first deep
line, which used to be the first time the old code noticed the buffer was too
small and had to reallocate. A debug_assert now also checks that the proven
bound actually holds every time (compiled out of release builds entirely, so it
costs nothing there, but still catches a violation in any debug run or test).
Reserving the capacity up front only changes when the buffer is allocated,
never what the search computes, so every solve value is unchanged.


ONE STEP BEFORE THE BAG IS EMPTY: THE ONE-KNOWN-BAG-TILE CASE

Everything so far only ever solves once the bag is already empty. This adds a
bag holding exactly one tile, where that tile's identity is passed in as
already known.

The rules at that point are simple enough to solve exactly with a small extra
layer on top of the empty-bag solver, never touching its search logic. Placing
any tile always draws that single bag tile back (a play draws min(placed, 1) =
1 whenever it places at least one tile), refilling the mover's rack -- so a
play here can never empty the mover's rack, and there is no play-out to worry
about while this one tile remains. After the draw, the position is a plain
empty-bag endgame with the opponent to move, so the existing solver handles it
unchanged. Passing draws nothing, leaving the one tile in the bag; two
consecutive passes end the game with that tile staying unseen and unscored for
either side.

solve_one_in_bag(player_idx, bag_tile) runs this shallow bag-phase minimax: for
the mover's own turn only, try every legal play, draw the known tile (landing
on the rack as an undesignated blank if that tile is a blank -- its letter is
only chosen once it is actually played), and hand the resulting fully-known
position to one reused empty-bag sub-solver; or pass, and recurse one ply with
the roles swapped. Tests add a trusted, independently-written plain-negamax
reference for this one-in-bag case and check the fast solver against it for
both movers and several different bag tiles, plus two hand-checkable positions
(a forced double pass, and a play that must draw before it can end). The
empty-bag solver's own differential check is unchanged.


WHEN THE MOVER DOESN'T KNOW WHICH TILE IS LEFT: PEG

The one-known-bag-tile solver assumes the mover already knows which tile is
sitting in the bag. In a real game that is almost never true: the mover only
knows which tiles are UNSEEN -- not on the board and not on their own rack --
and the single bag tile could be any one of them, with the rest making up the
opponent's rack. This situation, one tile left in an otherwise fully-known
position, is what this file calls PEG (pre-endgame): not yet an ordinary
empty-bag endgame, but one move away from becoming one.

solve_peg_one_in_bag solves it exactly rather than by sampling: for every
distinct unseen tile, try it as the hypothesis for the bag tile (the remaining
unseen tiles then form the opponent's rack, making that hypothesis fully known)
and hand it to the one-known-bag-tile solver. Each hypothesis is weighted by
how many copies of that tile are unseen -- a tile with three copies still
unseen is three times as likely to be the one actually in the bag as a tile
with only one copy left. peg_aggregate then folds all the weighted hypotheses
into a single win rate (a draw scoring half a win) and a single expected point
margin, both from the mover's point of view. Full enumeration, not sampling:
every distinct unseen tile is tried, so the mover's uncertainty about the bag
is modeled exactly, not estimated.

This enumeration weighs only plays and passes, so it is correct only where an
exchange is not an option. With a single tile in the bag an exchange is legal
only when the config permits exchanging that few tiles (Spanish does; English
and most others require a fuller bag), and there the mover could exchange
instead. Rather than report a number that silently assumes the exchange away,
solve_peg_one_in_bag returns a result that DECLINES such a config with an
honest "exchange not modeled" -- every other config is still answered exactly.
The exchange case is left unsolved here.

The solver's scope is exactly zero or one tile in the bag. One tile is the
PEG-1 case -- the one-in-bag pre-endgame this section solves; zero tiles is
already an ordinary empty-bag endgame. A bag holding two or more tiles is a
larger pre-endgame -- PEG-2, PEG-3, and larger -- and each extra tile in the
bag multiplies the draws the mover must weigh. Those larger pre-endgames
are a genuine generalization this file does not attempt, not merely an input it
rejects: such a position falls through to the same "not endgame yet" error the
CLI raises for any position with too many tiles left unaccounted for. Only
PEG-1 (and the plain empty bag) is in scope here.


SOLVING A REAL POSITION FROM THE COMMAND LINE

Everything above could only be exercised through the built-in demo positions or
a harvested-corpus batch file. A minimal command line now takes a lexicon, a
board (in FEN, the same compact board notation the rest of wolges uses), and
the mover's own rack, works out the opponent's tiles from whatever is unseen,
and runs whichever solver actually applies: an ordinary empty-bag endgame, or
the one-in-bag PEG enumeration when exactly one tile is left in the bag. The
batch mode and the built-in demo positions are unchanged.

By default the result covers only the tiles still to be played: the win, draw,
or loss and the point margin describe the rest of the game, not the game so
far. An optional trailing number says how many points the mover is ahead of the
opponent right now (negative if behind), and it is added to the solver's margin
so the reported outcome reflects the whole game rather than just the value of
the remaining plays.

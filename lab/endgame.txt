THE ENDGAME SOLVER: EXACT PLAY ONCE THE BAG IS EMPTY

Every leave table in this repo (lab/leavegen.txt) values a KEPT rack by an
estimate, because the rest of the game is still unknown -- more tiles will be
drawn, the opponent's rack is hidden. Once the tile bag is empty, neither of
those is true any more: every tile still in play is either on the board or on a
rack, so with both racks known the whole rest of the game is just a finite,
fully-visible sequence of plays and passes. Leave values stop applying at that
point -- lab/leavegen.txt notes that with an empty bag no tiles are drawn, so
the leave table is not consulted there; this file is about solving that fully-
visible tail EXACTLY, by negamax game-tree search (search every legal reply, at
each ply picking whichever line values best for the player to move, alternating
who that is) with iterative deepening (search one ply deeper each pass, using
the previous pass's answer to speed the next one up) all the way to the game's
end.

evaluate(player_idx) is the entry point that prints the search's per-depth
progress. Callers that want the answer as DATA rather than printed lines get
two additions here: solve(player_idx) runs the identical search silently and
returns just the final converged value, and collect_pv(player_idx, out) walks
the best line the search found and pushes each ply's (equity, play) pair into a
caller-owned, reusable vector, so the caller can keep that line around after
the solver itself goes out of scope. Both share the same core loop as
evaluate() -- pulled out into one private routine with a verbose flag -- so
this is a pure refactor: no search logic changes, and evaluate()'s printed
output is exactly as before.


A REAL VALUE FOR THE DEPTH-LIMIT LEAF

Iterative deepening stops each pass at a fixed depth and returns whatever value
the search assigned to the positions at that depth limit -- the "leaves" of
that pass's search tree (a tree-search leaf; see lab/leavegen.txt for the very
different sense of "leave", the kept tiles after a play). Until now a
depth-limited leaf returned a placeholder zero rather than a real value. It now
returns the leftover-tile point margin the position would score if the game
ended right there (both players passing), from the side to move's perspective
-- the same kind of terminal scoring the rest of the solver already uses when
the game genuinely does end.

Today's converged answer does not change: the search always deepens past a
depth-limited line before it actually stops there (an existing guard already
guarantees the search keeps going rather than settling for a capped-depth
value), so no real search run today ever returns this placeholder as its final
answer. But a depth-limited leaf is still a real, reachable position, not a
sentinel that should never be valued -- it deserves its correct value
regardless of whether today's search ever stops there.


BATCH-SOLVING HARVESTED POSITIONS

autoplay can now harvest the bag-empty positions it plays through into a file.
An opt-in batch mode (set WOLGES_ENDGAME_BATCH to that file's path) reads the
harvested positions back, prunes each one's word list to what is reachable,
solves each with solve() (silent and deterministic, from the previous entry),
and prints only each line's index and its value -- nothing else, so a whole
run's output can be hashed or diffed byte for byte. That makes it easy to
bulk-time the solver over many real positions, or to catch an accidental value
change in one pass. With the env var unset, the existing hardcoded demo
positions run exactly as before.


TWO SEARCH SPEEDUPS

Alpha-beta search (negamax with a window of values it can stop early outside)
prunes a branch as soon as it proves that branch cannot beat what the search
has already found elsewhere. Both speedups here only change how fast the search
reaches the SAME answer, never what that answer is -- alpha-beta returns the
same value no matter how much of the tree it prunes away.

Iterative deepening had been re-searching every depth from scratch with the
widest possible window ((negative infinity, positive infinity)). Aspiration
windows narrow that: from the second depth on, search first in a narrow band
around the PREVIOUS depth's value, since the true value almost always lands
close to it, which lets alpha-beta prune far more of the tree; only if the
narrow search's result actually falls outside that band does the search widen
and try again, so the final value is exactly what the full window would have
found -- a modest wall-clock win, larger the deeper a position's search goes.

Move ordering is the order a search tries a node's legal moves in -- trying the
best ones first lets alpha-beta prune the rest sooner, so most nodes stop after
a handful of moves once a cutoff fires. That makes fully sorting every node's
whole move list on every visit look wasteful: lazy move ordering brings only
the best few moves to the front and sorts the remainder only if the search
actually runs past them. This is value-identical by construction -- alpha-beta
returns the same result regardless of the order moves are tried in -- but for
bag-empty endgames it turns out to be a slowdown, not a speedup: the move lists
are short and iterative deepening revisits each node many times, so the full
sort is already cheap and the extra partial-select-then-tail-sort step is pure
overhead, about 7% slower here at every cutoff point tried. It ships off by
default behind a const, kept as an opt-in study lever for wider or deeper
position mixes where trying fewer moves first might still pay off.

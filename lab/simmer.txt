THE SIMMER: PICKING A MOVE BY ROLLING OUT GAMES

Every other file in lab/ values a LEAVE -- the tiles kept after one play --
with a table built ahead of time. The simmer is a different kind of tool: it
picks a MOVE, right now, on a real board, by actually playing out the rest of
the game (or a fixed number of turns of it) many times from each candidate play
and seeing which one comes out ahead on average. It is a 2-ply Monte Carlo
rollout move picker: ply 0 is the candidate itself, ply 1 is the opponent's
reply (found by the ordinary move generator, one play each), and so on for as
many plies as the caller asks for. "Simmer" is short for simulate; a Monte
Carlo method is one that estimates an answer by running many random trials and
averaging, rather than computing it exactly.

The core piece is a per-instance ChaCha20 random-number generator, reseed(seed)
for a reproducible run (the same seed and settings replay the same draws no
matter how many threads run them), and a SimmerConfig struct that collects the
model's tunable numbers: whether spreads are read in whole points or raw
millipoints (equity::SCALE = 1000; see lab/leavegen.txt), the win-probability
weightage for a game that can or cannot be played out within the simmed plies,
and the sigmoid consts described below. new() seeds from system entropy so an
un-reseeded caller keeps its original non-reproducible behavior; only a caller
that opts into reseed() gets a repeatable stream.

Two small helpers are pulled out so every caller shares one definition. The
unfinished-game win probability is a hand-tuned sigmoid of the score margin: a
lead of (sigmoid_spread + tiles_left_unseen) points reads as sigmoid_prob (a
config knob, default 90%), calibrated so a bigger lead is worth more confidence
the further along the game is and less when many tiles remain unseen. The
per-candidate objective a rollout reports is the spread (in points, unless the
config's descale flag is off) plus the win probability scaled by its weightage
-- one function, sim_objective, used by both the interactive move picker and
the batch harness, so they can never drift apart.


DROPPING THE ASYNC RUNTIME

The move picker's earlier shape ran its rollout loop alongside an async
wall-clock timer (a tokio task that woke up on a schedule and set a stop flag),
which meant the whole picker had to run inside a tokio async runtime just to
host that one timer. Once the timer is replaced by a fixed iteration budget (a
plain count of rollouts to run, not a clock), the timer task, the runtime, and
the unsafe "this type is safe to send between threads" assertion that came with
it are all dead weight -- with the async machinery gone the picker's fields are
all themselves safe to send, so the compiler can derive that property instead
of a human asserting it. Stripping tokio out entirely is also what lets the
picker build for a single-threaded wasm target, which cannot host a tokio
runtime at all. Picking a move is now a plain synchronous function call.


MEASURING THE SIMMER: A TWO-SEAT COMPARE

lab/leavegen.txt's english-compare measures leave TABLES: each seat picks its
move by one greedy look at the board with its own table. english-sim-compare is
the simmer's counterpart: both seats share the same leave table, but each picks
its move by running the full rollout instead of a single greedy look, and the
two seats can each be configured differently. Every seat reads its own settings
from WOLGES_SIM_P0_* and WOLGES_SIM_P1_* (today just DESCALE, on or off), and
both share one fixed WOLGES_SIM_ITERS rollout budget per move. Giving both
seats identical settings and expecting the result to land near 50% is a harness
sanity check (an unbiased comparison should tie); giving them different
settings measures which one plays better.

As in english-compare, each game pair starts from the same deal with the seats
swapped, so first-move advantage cancels between the two halves of the pair.
Each seat's simmer is reseeded before every single move from a mix of the run's
seed, the pair number, the game number, and the turn number, so the whole run
-- board positions, every rollout's random draws, the final result -- is a
deterministic function of the seed alone: replaying the same seed at any thread
count reproduces the same numbers exactly.


SPENDING THE ROLLOUT BUDGET WHERE IT MATTERS: THE ADAPTIVE ALLOCATOR

The picker's default way of spending its fixed rollout budget is round robin:
every surviving candidate is rolled out on every iteration, until the
confidence-interval prune (lab/leavegen.txt-style: mean plus or minus a margin
that narrows as samples grow) drops the clearly-worse ones. Early on, before
any pruning has happened, that spends as many rollouts on plays that are
obviously bad as on the two or three that might really be the best -- the
budget is not concentrating on the decision that matters.

The Adaptive allocator instead rolls out, each iteration, only the current
leader (the highest running mean), the strongest challenger (the non-leader
whose gap to the leader, measured in standard errors, is smallest), and one
least-sampled candidate as a floor. The floor matters because the prune needs
every candidate to already have some samples -- a candidate with no rollouts
yet has no confidence interval to compare, and would either dodge the prune
forever or be dropped instantly for the wrong reason. Because of that floor
requirement, Adaptive actually runs as round robin at first, until the very
first prune, then switches to leader/challenger/floor.

This changes only WHICH candidates get rolled out between prunes; the prune
rule itself (drop anyone the leader's confidence interval has already cleared)
is unchanged. It is opt-in: Allocator::RoundRobin stays the default (byte-
identical to before), Allocator::Adaptive is set via set_allocator and wired
into english-sim-compare per seat as WOLGES_SIM_P0_ALLOCATOR / P1_ALLOCATOR, so
the two can be measured against each other.


STOPPING EARLY ONCE THE LEADER IS CLEARLY BEST

Left alone, the picker always spends its whole iteration budget (short of
pruning down to a single survivor). On a lopsided position that keeps rolling
out long after the answer is obvious; on a close one the fixed budget may not
be enough to be sure at all. The confidence stop adds a second way to finish a
decision: at every prune boundary, if the leader's lower confidence bound
already clears every other survivor's upper confidence bound, stop right there
-- the remaining budget becomes a safety cap rather than a requirement.

Comparing one leader against every other survivor, and repeating the check at
every prune, means a plain single 95%-confidence threshold would let the chance
of picking the wrong leader compound over all those repeated comparisons. The
stop corrects for the number of arms with a union bound (the rule that the
chance of ANY of several unlikely events happening is at most the sum of their
individual chances): the Gaussian tail bound P(Z > z) is at most exp(-z^2 / 2),
so splitting an overall error budget of delta evenly across the (num_survivors
- 1) comparisons and solving for z gives z = sqrt(2 * ln((num_survivors - 1) /
delta)). This correction is for the number of candidates being compared, not
for how many times the stop is re-checked as rollouts accumulate; the fixed
iteration cap is what bounds that repeated peeking.

It is opt-in: StopRule::FixedCap (the default) always runs the full budget,
unchanged; StopRule::Confidence, set via set_stop_rule with set_stop_delta for
the target overall error, stops early once separated. Wired into
english-sim-compare per seat as WOLGES_SIM_P0_STOP / P1_STOP and
WOLGES_SIM_P0_STOP_DELTA / P1_STOP_DELTA.


A MUTABLE WORKING SET FOR STUDY SESSIONS

Everything so far runs one decision start to finish. A study session (a human
or a tool poking at a position interactively) wants more: bring in a move the
picker generated but never rolled out, or bring back one the prune already
dropped. Every candidate now carries a stable stream id (distinct from its
position in the list, which shifts as candidates are added or pruned), so a
retired candidate can be named later even after the list has been reshuffled.
Three operations use that id: add_play brings an already-generated play into
the working set as a fresh candidate with zero samples; readmit_with_history
brings a retired candidate back WITH its accumulated rollout statistics (its
old samples still count); readmit_fresh brings the same play back but throws
its old statistics away and starts it at zero. retired_stream_ids and
stream_count let a caller see what is sitting retired and check how many
rollouts a given candidate already has.

Candidates share each iteration's opponent draw (common random numbers, a
variance-reduction technique: comparing candidates on the SAME hidden tiles
removes noise that would otherwise come from comparing them on different
draws), but each candidate's own rollout draws from a private snapshot of that
shared state, so one candidate's tile exchanges never shift another's draws.
Readmitting a retired candidate is exactly as safe as any other candidate in
the working set -- there is nothing special to isolate.

None of this touches ordinary batch play: pick_a_move never calls these
operations, so its output is unchanged. english-sim-mutate-check is a headless
self-check confirming a readmitted candidate really does keep its statistics
across the round trip.


LOOKING IN ON A DECISION WITHOUT DISTURBING IT

The confidence stop and the ability to resume a decision in pieces already let
a study session spread one decision's rollouts over several calls. What was
still missing was a way to check on a decision between those calls without
changing anything. best_so_far reports the current leader's play index (the
candidate with the highest running mean right now); is_decided reports whether
the confidence stop would fire at this instant, at the currently configured
target. Both are read-only -- calling them runs no rollouts and changes no
state.

Together with set_num_sim_iters (the per-decision rollout cap) and
set_stop_delta (the confidence stop's target error, described above), a caller
now has everything it needs to budget one decision from the outside: cap the
work, set how sure the stop must be, then between resume calls poll best_so_far
and is_decided to pause, look at the current leader, and decide whether to keep
going or stop -- instead of only being able to run the whole budget blind or
abandon the decision outright.


ONE ENGINE, NOT TWO: FOLDING THE OBSERVABLE STUDY LOOP IN

The interactive terminal driver (main_simmer) had grown its own copy of the
rollout loop, separate from the real move-picker engine described above, so it
could track each candidate's running equity and win rate and print a
leaderboard as a study session progressed. Two loops meant two behaviors that
had to be kept in step by hand, and any future change to the real engine (a new
allocator, a new stop rule) would need to be re-implemented in the terminal
copy to actually show up there.

The engine itself now has an opt-in observe mode. run_iterations already
computes each rollout's spread and win probability on the way to the objective
described earlier; with observe on, it also feeds those two numbers into
per-candidate running equity and win-rate statistics, and a leaderboard method
ranks every candidate -- active or retired -- by mean. With observe off, the
default, none of that extra bookkeeping runs, and ordinary batch play is
byte-identical to before.

The terminal driver is now a thin shell: build one engine with observe on, run
the rollout budget through the same begin_decision / resume calls a study
session would use, and print the leaderboard as the estimates converge. The
separate copy of the loop is gone.


AN EMPIRICAL WIN% TABLE AS AN ALTERNATIVE TO THE SIGMOID

The unfinished-game win probability described earlier is a hand-tuned sigmoid
of the score margin -- a formula, not a measurement. Where enough self-play
games have already been recorded, an empirical table of P(win | margin, tiles
left), built from counting how those recorded games actually turned out, may be
better calibrated than a hand-tuned formula.

WinProbSource picks which one a config uses: Sigmoid (the default, unchanged)
always uses the formula; Table looks the terminal position up in a WinPctTable,
by margin and by how many tiles are left in the bag and racks, and falls back
to the sigmoid for any margin-and-tile-count combination the table never saw
enough of to have an entry. The table is handed to the simmer by reference and
defaults to none, so a caller that never asks for Table sees no change at all.

english-sim-compare can give each seat its own source (WOLGES_SIM_P0_WINPROB /
P1_WINPROB) sharing one table (WOLGES_SIM_WINPCT_TABLE), to weigh the sigmoid
against the empirical table directly, seat against seat.


OPTIONAL DETERMINISTIC PARALLEL ROLLOUTS

A single decision's rollouts are independent of each other, so on a native
(non-wasm) build they can run across several threads at once. The opt-in
parallel mode splits one decision's iteration block into contiguous slices, one
per thread, each thread running its own simmer instance and returning that
slice's per-iteration values for the caller to combine.

Splitting work across threads would normally make the result depend on how many
threads happened to run it and in what order they finished, which breaks the
"same seed, same result" guarantee the rest of this file relies on. Two choices
restore it. First, each iteration's draw is seeded from the decision's own seed
mixed with that iteration's index, rather than flowing from one shared stream
-- and the simmer's prepared starting position is restored before every
iteration -- so an iteration's draw depends only on its own index, never on
which thread ran it or what ran before it. Second, the iterations' values are
reduced in one fixed ascending order afterward, not by combining each thread's
own running statistics: floating-point variance combination is not associative
(a + b then + c can differ in its last bits from a then + b + c when + is
inexact), so combining by thread would make the result depend on the thread
count in ways too small to see by eye but able to flip which candidate wins a
close decision. Reducing every iteration's raw value in the same fixed order
regardless of which thread produced it removes that dependency: two threads and
eight threads now give byte-identical results.

Gated off entirely for the single-threaded wasm build (which has no threads to
split across) and off by default even where threads are available (one thread
keeps the original single-stream behavior, byte-identical to before this knob
existed); english-sim-compare exposes the thread count as
WOLGES_SIM_DRIVER_THREADS.
